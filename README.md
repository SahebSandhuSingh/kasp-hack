# AI Representation Optimizer

> **Hackathon Project** — A tool that simulates how AI shopping agents perceive and evaluate a Shopify store, shows merchants why their products get skipped, and tells them exactly what to fix.

---

## 🧠 The Problem

AI-powered shopping agents (like those built on LLMs) are increasingly being used to recommend products to users. These agents don't browse stores like humans — they evaluate structured product data. If a product has a vague description, no return policy, or zero reviews, an AI agent will skip it — and the merchant has no idea why.

**AI Representation Optimizer** solves this by simulating the agent's decision-making process, exposing which products are invisible to AI, diagnosing why, and proving that fixing the data fixes the visibility.

---

## ✅ What's Built (Steps 1–4)

### Step 1 — Single Query Simulation (`simulate.js`)
Loads a mock Shopify store (12 protein bar products with intentionally varied data quality), runs a single user query through the Groq API, and returns 2 selected products with reasons + all rejected products with specific diagnostic rejection reasons.

### Step 2 — Counterfactual Re-run Engine (`rerun.js`)
Runs the simulation with original product data, injects an "improved" version of a rejected product (`improve.js`), re-runs the simulation, and prints a structured before/after delta proving that fixing the data moves a product from rejected → selected.

### Step 3 — Multi-Query Audit Engine (`audit.js`)
Runs 7 different queries across the full store, computes per-product **inclusion rate**, diagnoses underperforming products using rejection reason pattern matching, outputs a ranked merchant priority action plan, and saves `audit_report.json` for the frontend.

### Step 4 — React Dashboard (`/frontend`)
A Shopify Polaris-inspired merchant dashboard with three views:
- **Dashboard** — audit report with stat cards, ranked priority action plan with progress bars, issue pills, and top performers
- **Simulate Query** — live query input with suggested chips, real AI results, and highlighted diagnostic keywords in rejection reasons
- **Before & After** — p7 counterfactual with side-by-side data quality cards, delta table, and improvement verdict

## 🚀 Setup & Usage

### Prerequisites
- Node.js (v18+)
- A [Groq API key](https://console.groq.com) (free tier works)

### 1. Install dependencies

```bash
# Backend
cd backend && npm install

# Frontend
cd ../frontend && npm install
```

### 2. Configure your API key

Edit `backend/.env`:
```
GROQ_API_KEY=your_groq_api_key_here
```

### 3. Generate the audit report (required before running the dashboard)

```bash
cd backend
node audit.js
```
> ⏱️ Takes ~2 minutes (7 API calls with 15s delays to stay under rate limits). Saves `audit_report.json`.

### 4. Start the backend server

```bash
cd backend
node server.js
# → Running on http://localhost:3001
```

### 5. Start the frontend

```bash
cd frontend
npm run dev
# → Running on http://localhost:5173
```

## 🖥️ API Endpoints (server.js)

| Method | Endpoint | Description |
|---|---|---|
| `GET` | `/api/audit` | Serves `audit_report.json` to the dashboard |
| `POST` | `/api/simulate` | Body: `{ query }` — runs a live simulation against all 12 products |
| `POST` | `/api/rerun` | Runs the before/after counterfactual for p7, returns `{ before, after, delta, verdict }` |

---

## 🖥️ CLI Scripts (backend only)

| Command | Description |
|---|---|
| `node simulate.js` | Run a single hardcoded query, print results |
| `node rerun.js` | Run the p7 counterfactual, print before/after delta |
| `node audit.js` | Run 7 queries, print full audit report, save `audit_report.json` |
| `node server.js` | Start the Express API server |

| Product | Inclusion Rate | Priority |

| ProForge Whey Protein Bar | 85.7% | ✅ PERFORMING WELL |
| IronMax High-Protein Crunch Bar | 71.4% | 🟢 MINOR FIXES |

| PowerZone Protein Bar | 0% | 🔴 CRITICAL |
| GreenLeaf Organic Oat Bar | 0% | 🔴 CRITICAL |
| ZenProtein Ayurvedic Bar | 0% | 🔴 CRITICAL |

**Key insight:** Products with null `return_policy`, null `ingredients`, vague descriptions, and fewer than 20 reviews are consistently invisible to AI shopping agents — regardless of their actual product quality.

## 🔍 Issue Flags Explained

| Flag | Triggered When |
|---|---|
| `missing_return_policy` | Rejection reason mentions "return" |
| `missing_shipping_info` | Rejection reason mentions "shipping" or "delivery" |
| `weak_social_proof` | Rejection reason mentions "review" or "trust" |
| `weak_description` | Rejection reason mentions "description", "vague", or "promotional" |
| `missing_ingredients` | Rejection reason mentions "ingredient" |
| `price_issue` | Rejection reason mentions "price", "budget", or "expensive" |

## 💡 Core Insight

> AI shopping agents are trained to be skeptical. Missing data isn't neutral — it's a rejection signal. A product without a return policy, clear shipping, or real reviews looks **untrustworthy** to an LLM, even if the product itself is excellent. This tool makes that invisible filter visible.
